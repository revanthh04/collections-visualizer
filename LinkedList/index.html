🔗 Node-Based Storage

Unlike an ArrayList, a LinkedList does not store elements in contiguous memory. Instead, it creates a wrapper object (a Node) for each element:

  • Scattered Memory: Nodes live wherever there is free space in the heap.
  • Doubly-Linked: Each node holds 3 things: a reference to the prev node, its item, and a reference to the next node.

⚡ The O(1) Operations

Because it holds pointers to the first (head) and last (tail) elements, endpoint operations are incredibly fast:

  • Adding/Removing at Ends: addFirst(), addLast(), removeFirst(), and removeLast() take exactly O(1) time. They only require updating a few pointers.
  • This makes it excellent for Queue and Deque implementations.

📉 The Traversal Cost

LinkedList loses the O(1) random access advantage of arrays:

  • No Direct Indexing: To find get(i), it must iterate node-by-node from the head (or tail), taking O(N) time.
  • Worse Cache Locality: Because nodes are scattered, the CPU cache will miss frequently, making iteration slower in practice than an ArrayList.
  • Memory Overhead: Every item requires an extra Node object with pointers.

⚙️ Common Methods & Time Complexity

Method Description Origin Time Comp.
void addFirst(E e) Inserts element at the beginning. Used in Stack (push)/Deque. Deque O(1)
void addLast(E e) / add(E e) Appends element to the end. Used in Queue/List. Deque/List O(1)
E removeFirst() / poll() Removes and returns the first element. Used in Queue/Stack/Deque. Deque/Queue O(1)
E get(int i) Returns element at index i by traversing node links. List O(n)
void add(int i, E e) Inserts at index i. Traversal is O(n), but insertion is O(1). List O(n)

* The visualizer demonstrates that finding the exact node takes O(n) traversal, but the actual operation of 'unlinking' and 'linking' nodes is instantaneous O(1), no shifting required.
